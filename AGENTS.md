## 7. Working Method

Before editing:

1. Read the relevant files.
2. Inspect the existing implementation.
3. Check for uncommitted or user-created changes.
4. Identify dependencies and risks.
5. Explain the intended outcome and files likely to change.

During implementation:

1. Make focused and reversible changes.
2. Reuse existing components and patterns where practical.
3. Avoid unrelated refactoring.
4. Preserve working analytics, metadata and integrations.
5. Keep business content separate from reusable presentation components where practical.
6. Do not overwrite or remove user work without explicit instruction.

After implementation:

1. Review the complete diff.
2. Run the available verification commands.
3. Test the affected user journey.
4. Report what changed.
5. Report verification results.
6. Identify anything that still requires CW's decision or manual action.

If a task is explicitly described as an audit, review or analysis, do not edit files unless CW also requests implementation.

## 8. Subagent and Parallel Work Rules

Use subagents only when the task contains independent workstreams that can be completed safely in parallel.

Suitable subagent work includes:

- Repository and architecture exploration
- Technical SEO review
- UX and conversion review
- Accessibility review
- Test and build-log analysis
- Product or competitor research
- Independent quality assurance

When delegating:

- Give every subagent a specific and bounded objective.
- Avoid assigning multiple agents to edit the same files.
- Use separate Git worktrees or branches for parallel implementation.
- Require subagents to distinguish facts from recommendations.
- Require concise findings rather than raw logs.
- Wait for all relevant subagents before consolidating the result.
- The main agent remains responsible for integration and final verification.

Do not delegate final pricing, brand positioning or commercial approval decisions.

Do not use subagents when a single focused agent can complete the task efficiently.

## 16. Git and Deployment Rules

Preserve existing user changes.

Before editing, inspect the current Git status.

Do not:

- Delete or overwrite unrelated work
- Rewrite Git history
- Force push
- Reset the repository destructively
- Commit directly to the main branch unless instructed
- Deploy to production unless explicitly instructed
- Include secrets in commits
- Remove configuration merely because its purpose is not immediately obvious

For substantial work:

- Use a focused branch or worktree.
- Keep commits logically scoped.
- Use clear commit messages.
- Review the final diff before recommending a merge.

Suggested commit style:

- `feat:` new user-facing functionality
- `fix:` defect correction
- `refactor:` internal change without intended behaviour change
- `docs:` documentation change
- `chore:` tooling or repository maintenance
- `test:` test additions or corrections

If the working tree already contains unrelated changes, work around them and report any overlap risk.
